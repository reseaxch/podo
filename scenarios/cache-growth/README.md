# cache-growth scenario fixtures

Canonical fixtures for the `cache-growth` incident. Produced by Workstream 4;
**owned and reviewed by Workstream 5** (per the fixture-ownership decision).
Consumers (adapters, evals, benchmarks, demo) read these files from here and do
not duplicate them.

## Files

| File | What it is | How it was produced |
| --- | --- | --- |
| `scenario.json` | Scenario descriptor and expected outcome. | Hand-authored. |
| `fixtures/graph.json` | System code graph of the demo monorepo (node-link JSON). | Graphify run over `demo/services` (`/graphify demo/services --directed --no-viz`). |
| `fixtures/telemetry.json` | Recorded OpenTelemetry-compatible event stream reproducing the incident (the "before" stream). | Deterministically generated by `generate-telemetry.ts`. |
| `fixtures/telemetry-after-fix.json` | **Candidate (pending WS5 acceptance).** Post-fix telemetry stream showing the remediated system (the "after" stream). | `generate-telemetry.ts --after-fix`. |
| `fixtures/graph.compat.json` | **Candidate (pending WS5 acceptance).** Strict compatibility manifest pinning `graph.json` to the public Graphify decoder. | Derived from the committed fixture and the public decoder (see below). |
| `generate-telemetry.ts` | Reproducible generator for both telemetry streams. | Committed so the telemetry can be regenerated/audited. |

## graph.json

Graphify node-link format: `nodes[]` carry `id`, `label`, `file_type`,
`source_file`, `source_location`, `community`; `links[]` carry `source`,
`target`, `relation`, `confidence` (`EXTRACTED` / `INFERRED` / `AMBIGUOUS`),
`confidence_score`, `source_location`. The versioned Graphify NetworkX decoder
(`decodeGraphifyNetworkxV1`) normalizes this fixture into the Podo code-graph
snapshot contract (`NormalizedCodeGraphSnapshot`), retaining each Graphify
confidence as `CodeGraphProvenance` (`extracted` / `inferred` / `ambiguous`).

Incident-critical nodes present: `CheckoutCache` (the unbounded cache), its
regression test, and the defect/rationale doc nodes.

## graph.compat.json (candidate)

> Status: **candidate**, pending Workstream 5 acceptance. A WS5-owned scenario
> asset, reviewed together with `telemetry-after-fix.json`.

A strict compatibility manifest that pins `graph.json` to the public Graphify
NetworkX decoder. It records **only verifiable data** — no invented Graphify CLI
version (none is available from recorded provenance, so it is omitted):

| Field | Meaning |
| --- | --- |
| `status` | `candidate` until WS5 accepts. |
| `fixture` | The fixture the manifest describes (`graph.json`). |
| `graphId` | Graph identity passed to the decoder (`cache-growth`). |
| `decoderVersion` | Public NetworkX decoder version (`GRAPHIFY_NETWORKX_DECODER_VERSION`). |
| `graphifySchemaVersion` | Graphify input schema version (`GRAPHIFY_SCHEMA_VERSION`). |
| `codeGraphSchemaVersion` | Normalized snapshot schema (`PODO_CODE_GRAPH_SCHEMA_VERSION`). |
| `graphCanonicalLfSha256` | SHA-256 of `graph.json` in canonical LF form (CRLF → LF before hashing). |
| `snapshotId` | Content fingerprint of the normalized snapshot the decoder produces. |

The fixture-integrity test re-derives every value from the committed fixture and
the public decoder and asserts an exact match. A decoder/version/schema mismatch
or any `graph.json` content drift **fails the gate closed** — the manifest must
be regenerated and re-reviewed whenever the graph, decoder, or schema changes.

## Regenerating the compatibility manifest

The manifest values are read directly off the public decoder and the committed
fixture (CRLF normalized to LF before hashing):

- `graphCanonicalLfSha256` = `sha256(graph.json with CRLF→LF)`;
- `decoderVersion` / `graphifySchemaVersion` / `codeGraphSchemaVersion` = the
  corresponding `@podo/plugin-graphify` exported constants;
- `snapshotId` = `decodeGraphifyNetworkxV1(graph.json, { graphId }).snapshot.id`.

If any of these change, regenerate `graph.compat.json` and re-submit it for WS5
review. No Graphify tool dependency is required to regenerate it.

## telemetry.json

A 22-event stream, every event conforming to the `TelemetryEvent` contract in
`@podo/contracts` (verified to normalize through
`@podo/domain` `normalizeTelemetryEvent`). Shape of the incident:

1. healthy heap baseline (~180 MB, `deploy-1041`);
2. defect deployment marker (`deploy-1042`);
3. monotonic heap climb after the deploy;
4. HTTP 500s with correlated traces and out-of-memory logs.

Fully deterministic (fixed base time, fixed 15s step, no randomness) so the
replay is reproducible per FR-2.

## telemetry-after-fix.json (candidate)

> Status: **candidate**, pending Workstream 5 acceptance. This is a WS5-owned
> scenario asset and should not be treated as canonical until WS5 adopts it.

The "after" side of a before/after comparison. It represents the same service
once the remediation has bounded the cache again. A 13-event stream:

1. post-fix deployment marker (`deploy-1043`);
2. bounded, flat heap samples (~180 MB, never climbing);
3. **no** HTTP 500 traces and **no** out-of-memory logs.

It runs on its own fixed timeline (base `2026-07-14T10:00:00Z`, same 15s step) so
its timestamps never collide with the incident stream. Interpreting the two
streams together: `telemetry.json` shows the incident (unbounded heap growth →
HTTP 500 → OOM), and `telemetry-after-fix.json` shows the remediated system
(bounded heap, no errors) — the observable evidence that the fix worked.

## Regenerating

```bash
# incident telemetry (before)
bun run scenarios/cache-growth/generate-telemetry.ts

# post-fix telemetry (after) — candidate
bun run scenarios/cache-growth/generate-telemetry.ts --after-fix

# graph (requires the graphify tool)
/graphify demo/services --directed --no-viz
cp graphify-out/graph.json scenarios/cache-growth/fixtures/graph.json
```
